# monthly_workflow.md

## 目的

本ファイルは、小規模法人がLLMを用いて月次の証憑整理・仕訳下処理・監査ログ生成を行うための簡易運用メモである。

本運用は、会計ソフトや税理士業務を代替するものではなく、月次で証憑・支出理由・仕訳候補・判断理由を失わないための下処理である。

---

## 1. 月次運用の基本方針

月次で行うことは、主に次の8点である。

1. 証憑を保護する。
2. LLMに仕訳候補と摘要を作らせる。
3. 支払履歴・銀行口座・カード明細と照合する。
4. 個人立替・未払い・未収を確認する。
5. 税区分・インボイス・電子取引データを確認する。
6. 月次サマリーを作成する。
7. 原本・処理用コピー・仕訳データを保存する。
8. 翌月への申し送りを残す。

---

## 2. 紙の領収書・レシート

紙の領収書・レシートは、購入時または帰宅後すぐにスマホで撮影する。

マスクが必要な情報がある場合は、画像加工アプリ等で墨消しした処理用コピーを作成する。

そのうえで、人間は原則として、領収書画像と最小限の摘要のみをLLMへ投入する。

人間が毎回、日付、購入先、金額、勘定科目、税区分、摘要、監査ログ等を手入力することは想定しない。

本運用の目的は、領収書情報の読み取り、仕訳候補の生成、摘要作成、監査ログ化をLLM側に担わせることで、人間側の入力負担を下げることにある。

標準的な投入例は次の通りである。

```text
添付画像の領収書を読み取り、以下の摘要を踏まえて、仕訳候補・摘要・監査ログを生成してください。

摘要：
事業用PCモニター購入。決済は個人カード立替。

仕訳＋摘要＋監査ログ化を依頼します。
```

LLMは、領収書画像をOCRで読み取り、人間が付記した摘要、月次チャット冒頭で共有された仕訳ルール、保存済みの会計処理方針を総合して、次のような形式で出力する。

```json
{
  "date": "2026-04-15",
  "vendor": "ヨドバシカメラ",
  "amount": 55000,
  "payment_method": "個人カード立替",
  "payer": "代表者",
  "category": "消耗品費",
  "tax_type": "課税仕入(10%)",
  "description": "事業用PCモニター購入。決済は個人カード立替。",
  "audit_log": "法人設立後の購入であり、開発業務に直接使用するため全額経費算入候補。証憑画像と摘要により事業関連性を確認。",
  "evidence_type": "紙領収書の撮影画像",
  "evidence_location": "",
  "notes": "最終的な勘定科目・税区分は月末照合時に確認する。"
}
```

画像加工が面倒な場合、または画像をLLMへ投入しない方がよい場合は、原本または処理用コピーをローカルに保存したうえで、人間が必要情報を文字で発話してログ化してもよい。

この場合のみ、次のような文字入力を行う。

```text
日付：2026-04-15
購入先：ヨドバシカメラ
金額：55,000円
内容：事業用PCモニター
支払方法：個人カード立替
用途：開発業務に使用
証憑：紙領収書をローカル保存済み

仕訳＋摘要＋監査ログ化を依頼します。
```

ただし、これは例外運用である。

原則は、人間が詳細テンプレを毎回作成するのではなく、領収書画像と最小限の摘要から、LLMが仕訳・摘要・監査ログ形式へ変換することである。

処理後の紙領収書は、月別の封筒またはファイルに入れる。

処理済みであることを示したい場合は、ペンで「済」と記載するか、済印を押す。

## 3. 電子領収書・カード明細・デビット明細

電子領収書、メール領収書、購入サイトの領収書、カード会社の利用明細、デビットカードの支払履歴は、可能な限りPDF・CSV・メール原本・スクリーンショット等で保存する。

LLMへ投入する際は、必要範囲のみを切り出す。

マスクすべき情報が含まれる場合は、LibreOffice Draw、画像加工アプリ、PDF編集ツール等で墨消しした処理用コピーを作成する。

画像添付が不要な場合は、証憑データを保存したうえで、次の情報だけを文字でLLMへ投入してもよい。

```text
日付
購入先
金額
支払方法
購入目的
事業利用割合
領収書・明細の保存場所
仕訳依頼
```

## 4. LLM仕訳依頼ルール

月別の税務申告下処理チャットを作成し、最初に account_setup_template.md の内容を投入する。

そのうえで、各証憑について、発話末尾に次のような意図文言を付ける。

仕訳＋摘要＋監査ログ化を依頼します。

LLMの出力形式は、保存済みメモリまたは account_setup_template.md のルールに従う。

標準出力項目は次の通りとする。

```json
{
  "date": "",
  "vendor": "",
  "amount": 0,
  "payment_method": "",
  "payer": "",
  "category": "",
  "tax_type": "",
  "description": "",
  "audit_log": "",
  "evidence_location": "",
  "notes": ""
}
```

## 5. 月末照合

月末に、次の照合を行う。

```text
紙領収書
電子領収書
カード明細
銀行口座
PayPay等の決済履歴
個人立替
役員借入金
LLM出力済み仕訳
ExcelまたはCSV管理表
```

確認することは、次の通りである。

```text
証憑はあるが仕訳がないものはないか。
仕訳はあるが証憑が見つからないものはないか。
個人カード立替が未精算のまま残っていないか。
友人立替・決済代行など特殊支払いが残っていないか。
金額・日付・購入先が一致しているか。
同じ支出を二重計上していないか。
```
---

## 6. 税区分・インボイス確認

各支出について、可能な範囲で次を確認する。

課税仕入か
非課税か
対象外・不課税か
軽減税率対象か
インボイス登録番号があるか
電子領収書・電子取引データを保存したか
帳簿上の摘要で後日説明できるか

不明な場合は、LLMに最終判断を丸投げせず、確認事項として残す。

## 7. 月次仕訳帳の生成

月末または翌月初に、当月のLLMログをもとに、Excelへ貼り付け可能な月次仕訳帳を生成する。

日次・随時処理でLLMへ投入した領収書画像、摘要、OCR結果、仕訳候補、audit_logは、あくまで「仕訳帳を作るための元データ」である。

これらを月末に整理し、当月分の仕訳帳として確定候補を作成する。

基本フローは次の通りである。

```text
領収書画像・電子証憑
↓
LLMによるOCR・仕訳候補・摘要・audit_log生成
↓
月別LLMログとして蓄積
↓
月末または翌月初に当月分を整理
↓
Excel貼付可能な月次仕訳帳を生成
↓
人間が確認・修正
↓
ExcelまたはCSVとして保存
```

LLMには、次のように依頼する。

```text
この月別チャット内で処理した当月分の証憑・仕訳候補・摘要・audit_logを整理し、Excelへ貼り付け可能な月次仕訳帳を生成してください。

列は以下としてください。

発生日
証憑日
借方勘定科目
借方税抜金額
借方税区分
借方消費税額
貸方勘定科目
貸方金額（税込）
支払総額（税込）
取引先
摘要
証憑種別
証憑保存場所
audit_log
確認状況
備考

未確定・要確認のものは、確認状況に「要確認」と記載してください。
```

出力形式は、Excelへ貼り付けやすいように、タブ区切りまたはCSV形式とする。

例：

```tsv
日付	借方勘定科目	借方金額	貸方勘定科目	貸方金額	取引先	摘要	税区分	支払方法	支払者	証憑種別	証憑保存場所	audit_log	確認状況	備考
2026-04-15	消耗品費	55000	役員借入金	55000	ヨドバシカメラ	事業用PCモニター購入。個人カード立替。	課税仕入(10%)	個人カード	代表者	紙領収書	2026年度/04月/01_original/	法人設立後の購入であり、開発業務に直接使用するため経費算入候補。	確認済	
```

月次仕訳帳を生成した後、人間が次を確認する。

```text
証憑が存在するか
金額・日付・取引先が一致しているか
二重計上がないか
個人立替・役員借入金の処理が合っているか
税区分に不明点がないか
未処理・要確認事項が残っていないか
```

確認後、月次仕訳帳をExcelまたはCSVとして保存する。

推奨保存場所は次の通りである。

```text
YYYY年度/
└── MM月/
    ├── 01_original/
    ├── 02_masked_for_llm/
    ├── 03_llm_outputs/
    ├── 04_monthly_journal/
    │   └── monthly_journal_YYYY-MM.xlsx
    ├── 05_monthly_summary/
    └── 06_notes/
```

月次仕訳帳は、期末に12か月分を統合して年次仕訳帳を作成するための中間成果物である。

期末に全証憑・全ログを一括でLLM処理しようとすると、取引数が数百件・数千件に増えた場合、トークン数、推論コスト、処理精度、文脈保持の面で限界が生じる。

そのため、本運用では、期末一括処理ではなく、月次で仕訳帳を生成・確認・保存する。

未確定事項や翌月確認事項は、月次仕訳帳および月次サマリーに申し送りとして残す。

期末処理では、12個のメンテ済み月次仕訳帳を人間が統合し、年次仕訳帳、B/S、P/L、申告下処理へ進む。

税務調査等で支出理由や判断過程を説明する必要がある場合は、月次仕訳帳だけでなく、対応する月別LLMログ、証憑画像、摘要、audit_logを参照する。

補足：税抜ベース管理を採用する理由

本モデルでは、免税事業者である期間を含め、将来的な課税事業者化や事業拡大への対応を容易にする観点から、税抜ベースでの記録・管理を推奨している。

これは税法上の取扱いを指示するものではなく、小規模事業者が将来的な事業成長や課税事業者化に備え、日常的な経営管理を容易にすることを目的とした運用上の設計方針である。

事業の成長に伴い、課税事業者への移行やインボイス制度への対応が必要となった場合、税込ベースのみで記録していると、過去データの再整理や消費税区分の見直しが発生しやすい。

一方、当初から税抜金額、消費税額、税区分を分離して管理しておけば、将来的な制度変更や事業規模拡大への対応が容易になる。

また、売上に含まれる消費税相当額と事業者固有の収益を意識的に区別する習慣は、資金管理上の健全性向上にも寄与する。

そのため本モデルでは、現時点で消費税申告義務が存在しない場合であっても、税抜金額・消費税額・税区分を併記した形で月次仕訳帳を生成し、将来の制度変更・事業成長・課税事業者化に伴う認知負荷と移行コストを低減するため、初期段階から継続的に税抜ベースで記録し管理することを推奨する。

なお、最終的な会計処理および税務申告については、適用される法令、会計方針、税理士等の専門家の助言に従うものとする。

## 8. 月次サマリー

月末に、LLMへ当月データを渡し、次のサマリーを作成する。

```text
当月支出合計
科目別支出
個人立替合計
未精算額
特殊処理一覧
証憑不足一覧
税区分不明一覧
翌月確認事項
```

このサマリーは、月次レビュー用であり、法定帳簿そのものではない。

---

## 9. 保存構造

チャット履歴を正本保存場所にしない。

正本は、ローカルまたは管理されたクラウドストレージに保存する。

推奨構造は次の通り。

```text
YYYY年度/
└── MM月/
    ├── 01_original/
    ├── 02_masked_for_llm/
    ├── 03_llm_outputs/
    ├── 04_excel_csv/
    ├── 05_monthly_summary/
    └── 06_notes/
```

原本とLLM投入用コピーは分ける。

原本
→ 法定保存・証拠保存用

処理用コピー
→ マスク後、LLM投入用

---

## 10. 翌月への申し送り

月末時点で未解決の事項は、翌月への申し送りとして残す。

```text
未取得の領収書
未精算の個人立替
税区分不明
インボイス不明
処理保留中の支出
専門家・税務署へ確認すべき事項
```
---

## 11. 注意事項

本運用におけるLLMの役割は、証憑整理、OCR補助、摘要作成、仕訳候補生成、audit_log生成、確認事項の抽出を支援することである。

LLMの出力は参考情報であり、会計帳簿、申告書、決算書等の最終確定情報ではない。

最終的な判断と責任は事業者自身に帰属し、必要に応じて税務署・税理士・専門家へ確認する。
